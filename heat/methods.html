The central question

Public heat communication in India runs mostly on dry-bulb air temperature. This dashboard asks: which cities look less dangerous on that ordinary ranking, but become high-risk once humidity, radiant heat, and time of day are accounted for? A prior-art review (see the repository's existing_tools_research.md) found serious tools for occupational heat stress, thermal-comfort datasets, and live weather maps, but no public India city-ranking tool built specifically around this dry-bulb-vs-humid-heat misranking. This dashboard exists to fill that specific gap — not to claim no one tracks heat risk in India, which official IMD/NDMA heat-wave communication clearly does.

Three metrics, kept strictly separate

These are different physical quantities with different risk meanings. This dashboard never compares wet-bulb values against WBGT-style thresholds, or vice versa.

1. Dry-bulb temperature

The ordinary air temperature, taken as-is from Open-Meteo. This is the public-conversation baseline this dashboard is contrasting against.

2. Wet-bulb temperature — Stull (2011) approximation

Wet-bulb temperature reflects how effectively the body can cool itself by sweat evaporation; unlike dry-bulb temperature, it accounts for humidity. Computed here via the empirical approximation published in:

Stull, R., 2011: "Wet-Bulb Temperature from Relative Humidity and Air Temperature." Journal of Applied Meteorology and Climatology, 50(11), 2267–2269. doi:10.1175/JAMC-D-11-0143.1

Formula (T = dry-bulb temp in °C, RH = relative humidity in %): 

Tw = T·atan[0.151977·(RH + 8.313659)^0.5]
     + atan(T + RH) − atan(RH − 1.676331)
     + 0.00391838·RH^1.5·atan(0.023101·RH)
     − 4.686035

Validity range: RH 5–99%, T −20°C to 50°C (accuracy degrades in the combined very-low-humidity / very-cold corner of that range, which this dashboard's cities never approach). Error bounds: −1°C to +0.65°C, mean absolute error under 0.3°C. The approximation assumes standard sea-level (101.325 kPa) pressure. Most of the ~50 cities here are low-elevation, but not all: for elevated cities — Srinagar (~1,600m), Bengaluru (~900m), and a few others in the 400–700m band — the wet-bulb values shown are slightly less exact (the formula modestly overestimates wet-bulb as pressure falls below sea level). The estimated WBGT is unaffected by this particular caveat: the Liljegren model takes each city's actual surface pressure as an input.

3. Estimated WBGT — Liljegren et al. (2008) method

Wet Bulb Globe Temperature (WBGT) is the internationally standard metric for outdoor occupational heat-stress risk (ISO 7243), used by militaries, sports governing bodies, and workplace-safety regulators worldwide, and applied by Indian researchers in urban and occupational heat-stress studies. It is not part of India's official heat-warning vocabulary — which is precisely the gap this dashboard examines: estimating WBGT from public weather data shows what a dry-bulb-only ranking leaves out. Unlike wet-bulb temperature alone, WBGT also incorporates radiant heat (solar radiation) and wind. True WBGT requires a physical instrument (a black globe thermometer) that this dashboard does not have access to. Instead, this is an estimate derived purely from standard meteorological variables (temperature, humidity, solar radiation, wind) — supplied here by gridded forecast/reanalysis data, not physical station instruments — using the method from:

Liljegren, J. C., R. A. Carhart, P. Lawday, S. Tschopp, and R. Sharp, 2008: "Modeling the Wet Bulb Globe Temperature Using Standard Meteorological Measurements." The Journal of Occupational and Environmental Hygiene, 5(10), 645–655. doi:10.1080/15459620802310770

The implementation here is a Python port of James Liljegren's original 1.1 C code (originally published by UChicago Argonne, LLC under contract with the U.S. Department of Energy), obtained via the MIT-licensed mdljts/wbgt R package wrapper. The final combining formula, taken directly from that source code: 

WBGT = 0.1·T_air + 0.2·T_globe + 0.7·T_natural_wet_bulb

Verification: rather than trust the port by inspection, the original C code (with its own demo driver) was compiled directly with gcc and run on two test cases (a hot/humid daytime scenario and a nighttime scenario). The Python port's output matched the compiled C binary's actual output to within about 0.02–0.1°C — consistent with ordinary floating-point precision differences between C's float and Python's double, not a translation error. Both test cases are kept as permanent automated regression tests in the repository (tests/test_liljegren_wbgt.py).

Caveat: this is an estimate from meteorological inputs, not a physical measurement. Liljegren-based estimates typically agree with measured WBGT to within about 1°C under normal conditions, worse in extreme hot-dry regimes.

Outdoor-work risk reference — NIOSH RAL/REL

This dashboard does not use a flat "safe/moderate/ danger" WBGT scale. An earlier draft did, using invented thresholds loosely modeled on ISO 7243 (<28°C / 28–30°C / 30–32°C / >32°C) — but checking those against the real standard found that ACGIH's and ISO 7243's actual tables vary continuously by workload and work/rest duty cycle, and are themselves paywalled (confirmed directly via OSHA's own Technical Manual, which states the ACGIH heat-stress tables "are copyrighted by ACGIH and is not publicly available"). Publishing invented numbers under a borrowed standard's name would have been dishonest, so instead this dashboard uses NIOSH's own public formula, from:

NIOSH, 2016: "Criteria for a Recommended Standard: Occupational Exposure to Heat and Hot Environments." DHHS (NIOSH) Publication 2016-106, Chapter 8.1 (equations, p.93–94) and Table 5-1 (cross-standard comparison, p.101). cdc.gov/niosh/docs/2016-106 

RAL [°C-WBGT] = 59.9 − 14.1·log10(M)   (unacclimatized workers)
REL [°C-WBGT] = 56.7 − 11.5·log10(M)   (acclimatized workers)

where M is the 1-hour time-weighted-average metabolic rate, in watts, for a "standard" 70kg/1.8m² worker doing continuous (60 min/h) work. This dashboard evaluates both formulas at a representative moderate workload (300 kcal/h ≈ 349W — the same value NIOSH's own worked example in the document uses), giving reference lines around 24.05°C (RAL) and 27.46°C (REL).

These are NOT flat "safe/danger" cutoffs. They vary continuously with workload, and depend on acclimatization, clothing, and work/rest scheduling — none of which this dashboard measures directly. Language on this site always says "lower-risk outdoor work windows under estimated WBGT assumptions," never "safe hours," and every threshold shown is labeled with the specific NIOSH reference it corresponds to.

Data sources

Live forecast

Open-Meteo Forecast API, hourly temperature, relative humidity, shortwave solar radiation, wind speed, and surface pressure, for all ~50 cities in a single batched request every 6 hours via GitHub Actions. No API key required; free tier is 10,000 calls/day, and this pipeline uses exactly one call per run.

1991–2020 historical normal

Open-Meteo's Historical Weather API (ERA5 reanalysis), pulled once per city (not on the live schedule) for 1991–2020, collapsed to per-calendar-date climatological means and maxima for both dry-bulb and wet-bulb temperature.

ERA5 is a global reanalysis on a roughly 25–31km grid — it is not a street-level station reading, and a real city's microclimate (urban heat island effects, a specific neighborhood's conditions) can differ from the grid-cell value shown here.

City list

50 Indian cities selected from Vynex/indian-cities-geodata (Apache 2.0 licensed; coordinates sourced from Census 2011 population figures and Google Maps), taking the largest cities by population with deliberate substitutions to ensure humid coastal and eastern cities are well represented rather than dominated by drier, better- known cities.

Map boundaries

India state/UT boundaries from DataMeet India community's "maps" repository (MIT licensed), simplified from 15.7MB to about 140KB using mapshaper for faster loading, with all boundary shapes otherwise unmodified. See the license/attribution file in the repository for full details.

What "misranking delta" means

Cities are ranked twice each day: once by today's peak (max) dry-bulb temperature, and once by today's peak estimated WBGT. The misranking delta is dry-bulb rank minus WBGT rank. A large positive delta means a city looks unremarkable on ordinary temperature (a high rank number) but is near the top of the humid-heat danger list (a low rank number) — a "climber," in this dashboard's language. This computation runs identically in the map and the rank-shift chart (both read from heat/js/data.js) so the two views can never disagree with each other.

The direction of the shift is not fixed, and this dashboard does not claim that humid coastal cities are always the underrated ones. Because estimated WBGT also weighs solar radiation and wind, there are days when inland northern cities climb and coastal cities fall. The claim being made is narrower and more defensible: a dry-bulb ranking misorders cities' physiological heat risk, and which cities it misorders changes with the weather — which is exactly why a live view exists instead of a one-off infographic.

AI-transparency note

This dashboard's data pipeline, scientific-formula implementations, and frontend code were built with AI assistance (Claude, Anthropic), working from a project brief and editorial decisions set by the site's author. Where that matters for trusting the numbers on this page:

  • Every formula was checked against a primary source (the original paper or standard, or a citation of it that could itself be verified) rather than accepted from a first-pass summary. Where a primary source could not be reached (the ACGIH/ISO 7243 exact tables, which are genuinely paywalled), that gap is stated above rather than papered over with an invented number presented as authoritative.
  • The Liljegren WBGT port was verified by compiling the original author's C code directly and comparing outputs numerically, not just by reading the ported code and judging it plausible.
  • All reproducible pipeline scripts (data fetching, the wet-bulb and WBGT calculations, the historical-normals builder) are in the public repository, along with an automated test suite and a running build log (BUILD_LOG.md) documenting what was built, what broke, and how each issue was found and fixed.
